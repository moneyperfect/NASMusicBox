# 我如何全程 Vibe Coding，把一个音乐软件从 MVP 做到真正可用

这是一个我最近做得非常开心，也非常有代表性的项目。

我做了一个叫 NAS 音乐器的本地音乐软件。它最开始只是一个能跑的 MVP：能搜索、能播放、能显示歌词、能做一点视觉效果，看起来已经像回事了。

但真正让我意识到“它开始变成产品了”的，不是某个功能做出来的那一刻，而是我开始连续遇到那种只有真实用户才会遇到的问题：

- 某首歌搜不到
- 歌词有时有，有时没有
- 桌面端和网页端表现不一致
- 功能明明已经修了，用户还是觉得慢
- 甚至到最后，源码明明已经提速了，用户实际打开的软件还是旧版本

这篇文章我不想写成“我用了哪些技术”，而更想写成一次方法论复盘：我到底是怎么靠一轮又一轮的反馈、排查和重构，把这个项目从“一个能跑的 demo”，打磨成“一个真的可以安装和长期使用的软件”的。

## 我为什么会做这个产品

我一直很喜欢那种“打开就能用”的软件。

不是那种看起来很厉害、但只能在开发环境里演示的软件；而是那种你双击打开、开始搜索、开始播放、开始收藏，甚至会愿意长期放在电脑里使用的产品。

NAS 音乐器最开始的想法其实很简单：我想做一个属于我个人品牌的音乐器。它不一定要一开始就做成一个巨大的平台，但至少应该满足几个很明确的条件：

- 我真的能用
- 它不是网页 demo，而是桌面软件
- 它有自己的品牌感
- 它能持续迭代

说白了，我不是想“写一个音乐播放器练手”，而是想把一个产品从 idea 一直推到可用状态。

## 一开始它只是个能跑的 MVP

最开始的版本，其实已经具备了一个音乐软件最基本的轮廓：

- 可以搜歌
- 可以播放
- 可以看歌词
- 可以做简单的视觉化

如果只是为了演示，那已经差不多够了。

但真正开始使用之后，我很快就发现一个很扎心的事实：

> 能跑，不等于能用。

比如，能搜到大部分歌不代表搜索真的可靠；能播出来不代表播放链路真的轻；能打包成桌面端不代表它已经像个正常软件；甚至功能都没问题，也不代表用户打开的是最新版本。

而产品和 demo 的分界线，恰恰就在这些地方。

## 真正难的不是功能，而是体验

这个项目让我越来越确信一件事：

> 做功能通常没有想象中难，真正难的是把体验做到“没有明显违和感”。

比如我这次遇到过几个非常典型的问题。

### 1. 搜索不是“能搜到”就算完成

一开始，搜索功能看起来是能用的，但很快我就遇到了一个很典型的失败案例：《夜に駆ける》搜不到。

这种问题最容易让人误判。你很容易先怀疑字符编码、前端请求、日文处理，甚至怀疑是不是网络偶发问题。

但真正的问题并不在“字”，而在“结果质量”。原来的链路太依赖 `yt-dlp` 搜索，只要第一批结果里混进不可播、年龄限制或者质量很差的内容，整个请求就会退化甚至失败。

后来真正有效的做法不是继续 patch 特殊字符，而是把搜索结果本身做干净：

- 跳过坏结果
- 去重
- 压低歌词版、KTV、翻唱和混剪
- 把搜索和播放解析拆开

一旦输入质量提高，后面很多问题都会一起变少。

### 2. 资料状态不能只留在前端

另一个明显的变化，是我从“前端临时状态”走向了“产品资料层”。

当产品开始有桌面端、长期使用、收藏、历史、下载、歌词偏移这些需求时，localStorage 很快就不够用了。

因为那时候这些数据不再只是 UI 状态，而是用户资产。

所以后来我把收藏、历史、搜索记录、下载记录、歌词偏移都统一收进了 SQLite。这一步看起来不如“新功能上线”那么显眼，但它其实非常关键。

因为从那一刻起，这个软件开始“记得住用户”，它才真正具备了长期使用的条件。

### 3. 桌面端不是包一层壳，而是切换使用场景

这也是我这次很深的一个感受。

很多项目把网页包成桌面端之后，会觉得“好，桌面版完成了”。但用户并不会因为你用了桌面壳，就自动把它当成桌面软件。

用户期待的是：

- 有托盘
- 有图标
- 有全屏
- 有迷你播放器
- 有安装包
- 有更新
- 菜单是中文的

所以我后面补的其实不是一个“技术封装”，而是一整套产品行为：

- 托盘和迷你播放器
- 全屏和隐藏
- 安装包和 GitHub Release
- 图标、品牌、中文菜单

当这些细节慢慢齐起来的时候，它才开始像一个软件，而不是一个项目。

## 这次最关键的一个难题：代码已经提速了，用户为什么还觉得慢？

如果只能选一个最值得写进这次复盘的时刻，我会选这个。

当时我已经做了一轮很重的性能优化。源码环境里，真正的慢点已经被拆出来了：

- 搜索元数据和播放解析被彻底拆开
- 搜索不再默认走最重的 `yt-dlp ytsearch`
- 点播放时只解析单个 `videoId`
- 直连媒体地址交给浏览器，`proxy-stream` 只做回退
- 歌词、搜索、解析和颜色提取都加了缓存

结果实测非常明显：

- 原来某些搜索链路能慢到两分钟以上
- 后来稳定回到了 2 到 3 秒级
- 点击搜索结果后的播放解析回到了 2.5 到 3.5 秒左右

按理说，这已经修好了。

但用户给我的反馈却非常直接：还是慢，而且是“特别慢”。

如果我当时继续沿着源码往下 patch，其实很容易越修越偏。因为这个时候最重要的信号不是“代码还有没有 bug”，而是：

> 用户的真实体验，和我的本地结论发生了冲突。

后来我去核对用户真正打开的路径，才发现一个非常关键的事实：

用户打开的是旧的 `dist/NASMusicBox/NASMusicBox.exe`。

也就是说，我优化的是源码和 GitHub 上的最新代码，但用户实际用的，仍然是前一天生成的旧桌面包。

这个发现特别有代表性，因为它提醒了我一件非常重要的事：

> 当真实体验和本地判断冲突时，先确认用户运行的是哪一份东西，而不是继续怀疑用户描述不准。

这是我这次觉得最有价值的一条经验。

## 我从这次 Vibe Coding 里真正学到了什么

很多人现在谈 Vibe Coding，容易把它说成一种“AI 帮我全自动生成”的浪漫叙事。

但我自己这次最大的感受反而是：

Vibe Coding 真正有价值的地方，不是它自动，而是它极大地缩短了“发现问题 -> 改动 -> 验证 -> 再修正”的循环。

真正有效的不是 AI 单独做了什么，而是这套协作方式：

- 我提出真实体验问题
- AI 快速分析、修改、验证
- 我继续从真实使用场景给反馈
- 再根据反馈继续修正

它更像是一个反应很快、不会嫌烦、可以陪你一直打磨细节的合作者。

如果你问我这次最重要的方法论是什么，我会总结成 5 句话：

1. 用户反馈优先级高于“我本地感觉没问题”
2. 先确认运行路径，再确认代码路径
3. 链路问题一定要阶段化计时
4. 结构性修复优先于单点 patch
5. 修复后必须回到真实使用环境验证

这几条不只是适用于音乐软件，几乎适用于所有你想从 MVP 走到可用产品的项目。

## 它为什么已经不再只是 MVP

如果只是从功能列表看，很多项目很早就能宣称“我做完了”。

但我现在越来越觉得，一个产品什么时候不再是 MVP，不是看它有多少页面、多少按钮，而是看它有没有跨过这些门槛：

- 数据能不能长期保存
- 桌面端能不能长期使用
- 用户能不能直接下载安装
- 出问题时能不能定位
- 体验是不是还存在明显违和感
- 文档和发布是否齐全

NAS 音乐器现在至少已经跨过了我心里的这条线。

它当然还远远不是一个“大而全”的成熟音乐平台，但它已经是一个：

- 可以安装
- 可以搜索
- 可以播放
- 可以收藏
- 可以下载
- 可以长期保留在电脑里使用

的软件了。

这对我来说，是非常不一样的阶段。

## 接下来我还会怎么做

如果继续迭代，我最看重的方向还是“个性化推荐”。

因为一旦有了推荐流，这个产品就会从“我主动搜歌的工具”，变成“我可以一直刷着听的产品”。

除此之外，我还很想继续做这几个方向：

- 电台模式 / 情绪模式
- 更完整的播放列表和收藏体系
- 下载资产管理
- 跨端同步
- 诊断与自检面板

尤其是最后一个，我现在反而越来越觉得值得做。

因为这次项目里最有价值的经验之一，就是很多问题其实不是没法解决，而是“没有被正确观察和拆分”。如果以后能把这些排查能力直接做成产品内的状态页，那这个软件会更像一个真正可维护、可持续迭代的产品。

## 结尾

如果让我用一句话总结这次经历，我会说：

> 我不是靠 Vibe Coding “自动做出了一个音乐软件”，而是靠它把一个原本只是能跑的想法，持续打磨成了一个真正可用的产品。

而这中间最重要的，不是某个技术选型，也不是某次生成结果，而是那种不断回到真实体验、不断承认问题、不断拆解链路、不断验证修复的过程。

我觉得，这才是这次项目真正值得写下来的地方。
