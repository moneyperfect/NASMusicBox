# NAS 音乐器复盘：从 MVP 到可用产品，哪些经验值得记下来

## 一句话结论

这次项目最有价值的收获，不是“做出了一个本地音乐软件”，而是验证了一套非常实用的产品打磨方法：当用户真实体验和本地结论冲突时，先不要继续改代码，而要先确认用户到底运行了什么、卡在链路的哪一段、以及你修复的到底是不是用户正在使用的那一份产物。

这套方法最后不只是把 NAS 音乐器从一个能跑的 MVP 做到了可用，还顺手搭好了桌面端、下载、歌词、发布、更新、安装包和后续可迭代的基础设施。

## 项目阶段变化

### 阶段 1：能跑的 MVP

最早的版本已经具备了“搜索、播放、歌词、可视化”的雏形，但它更像一个演示品：

- 有主要功能，但稳定性不够
- 有桌面壳，但不像真正的软件
- 有播放，但链路很重
- 有收藏和历史，但没有产品级的持久化和发布方式

这一阶段的目标是验证方向，不是验证体验。

### 阶段 2：从功能完整到产品完整

后续迭代补齐了这些真正决定“可不可用”的能力：

- 搜索容错和排序增强
- SQLite 本地资料库
- 多源歌词和歌词偏移保存
- 桌面端托盘、迷你播放器、全屏、安装包、自动更新
- 下载中心和 GitHub Release
- 中文化、品牌统一、桌面图标、说明文档

这时产品已经不是“一个项目”，而是“一个别人可以直接下载、安装、使用”的软件。

### 阶段 3：体验救援

最后一轮最关键的问题不是“有没有这个功能”，而是“用户为什么会感觉慢、而且慢得离谱”。

这一轮的结果最有代表性：

- 旧链路里，`search_youtube_entries("Yellow Coldplay")` 实测约 `151s`
- 单个视频播放解析一度约 `17.7s`
- 真正优化后，搜索稳定回到约 `2-3s`
- 点击搜索结果后的播放解析回到约 `2.5-3.5s`

这说明真正的分水岭，不是写了多少功能，而是有没有把真实使用路径里的结构性问题拆开。

## 最关键的 6 个难题

## 1. 搜索不是“能搜到”就够了，必须能稳定搜对

### 症状

- 某些歌曲能搜到，某些歌曲搜不到
- 典型案例是《夜に駆ける》
- 搜索结果里会混入歌词版、翻唱版、KTV 版、不可播视频

### 当时最容易出现的误判

- 以为是前端编码问题
- 以为是日文关键字处理有问题
- 以为是单纯换一个搜索词就能解决

### 真实根因

问题不在“字符”，而在“结果集质量”。原有链路把 `yt-dlp` 搜索结果直接拿来用，只要第一批结果里混进年龄限制、不可播或不合适的内容，整个请求就会退化甚至失败。

### 解决方式

- 给搜索结果增加容错和去重
- 明确压低歌词版、翻唱、KTV、混剪等结果
- 把搜索和播放解析分开
- 引入更轻量的 metadata provider，避免所有搜索都走重型 `yt-dlp ytsearch`

### 为什么有效

因为这次修的不是单首歌，而是搜索结果的“输入质量”。一旦输入质量提高，后续播放、歌词、下载都会一起稳定。

## 2. 收藏、历史、下载记录不能只放前端

### 症状

- 网页端和桌面端状态不一致
- 重启后数据容易散落在不同地方
- 收藏和历史只能算“临时记忆”，不适合作为长期使用的软件

### 当时最容易出现的误判

- 以为 localStorage 够用了
- 以为个人项目不需要资料层

### 真实根因

一旦产品进入桌面端和长期使用阶段，状态就不再只是 UI 状态，而是产品资产。收藏、历史、搜索记录、下载记录、歌词偏移都需要统一落盘。

### 解决方式

- 引入 SQLite
- 把收藏、历史、搜索、下载、歌词偏移统一放进后端资料库
- 前端只做展示和同步，不再做唯一真相来源

### 为什么有效

这一步让项目从 demo 变成了软件。因为“产品记得住用户”是长期使用的前提。

## 3. 歌词系统不能只依赖单一来源

### 症状

- 一部分歌曲没有歌词
- 即使有歌词，也不一定同步
- 有时候显示文案还很难看，比如“无法找到歌词？？？”

### 当时最容易出现的误判

- 以为歌词数据源本身不行
- 以为没有歌词就只能接受

### 真实根因

不是“有没有歌词”，而是歌词链路太单一。单一来源意味着一旦未命中，就只能失败。

### 解决方式

- 先走 `lrclib`
- 再尝试 YouTube 字幕 / 自动字幕
- 保存歌词偏移量
- 优化未命中文案和交互表现

### 为什么有效

这次修复的重点不是把歌词做得“更炫”，而是让歌词系统从单点依赖变成多路兜底。这样它才像产品能力，而不是附加功能。

## 4. 桌面端不是“包一层壳”就结束了

### 症状

- 托盘菜单、全屏、图标、快捷键、安装方式都不够产品化
- 用户虽然能打开，但很难把它当成熟悉的软件长期使用

### 当时最容易出现的误判

- 以为桌面端只是一个技术封装
- 以为浏览器能跑，桌面端就算完成

### 真实根因

桌面端真正解决的是“使用场景”，不是“运行方式”。用户对桌面软件的预期包括：

- 有托盘
- 有安装包
- 有图标
- 有全屏和隐藏
- 有中文菜单
- 有更新

### 解决方式

- 做正式桌面入口
- 做托盘和迷你播放器
- 做全屏模式和开机启动
- 做 Windows 安装包和 GitHub Release
- 统一中文菜单、品牌和桌面图标

### 为什么有效

因为用户不会把“技术上可运行的壳”当成产品，但会把“安装后像个正常软件”的程序当成产品。

## 5. 速度问题必须拆链路，不然永远是玄学优化

### 症状

- 用户反馈搜索和播放慢到接近不可用
- 本地感觉有些地方已经优化过，但体感仍然不对

### 当时最容易出现的误判

- 先怀疑前端
- 先怀疑网络
- 先做零散缓存
- 用“感觉快了一点”替代真实计时

### 真实根因

真正的瓶颈是：

- 搜索元数据和播放解析没有拆开
- `yt-dlp ytsearch` 被错误地放在主路径上
- 点播放时还在做一整套很重的搜索和解析

### 解决方式

- 搜索默认改成 `YouTube Data API -> ytmusicapi -> legacy yt-dlp`
- 点击结果后只解析单个 `videoId`
- 直接把媒体直链交给 `<audio>`
- `proxy-stream` 改成失败回退，而不是主路径
- 对搜索、解析、歌词、颜色提取加缓存和负缓存
- 把链路全部阶段化计时

### 为什么有效

因为这次修的是系统结构，而不是单个函数。真正让“分钟级等待”回到“秒级响应”的，不是某个小 patch，而是把“搜什么”和“怎么播”这两件事彻底分离。

## 6. 这次最有代表性的教训：代码修好了，不等于用户用到的是新版本

### 症状

- 本地源码版实测已经提速
- 用户反馈却仍然是两分钟搜索、一分钟播放

### 当时最容易出现的误判

- 以为用户描述不准确
- 以为还有隐藏代码路径没修
- 继续在源码上加 patch

### 真实根因

用户打开的是旧的 `dist/NASMusicBox/NASMusicBox.exe`，而不是最新代码生成的桌面包。也就是说，问题不在源码，而在产物。

### 解决方式

- 先确认用户实际打开的文件路径和生成时间
- 确认当前正在运行的是不是托盘旧实例
- 重打桌面包和安装包
- 要求完全退出旧实例再重启

### 为什么有效

这次最重要的不是“修掉了一个 bug”，而是验证了一条非常值得长期记住的原则：

> 当用户体验和本地结论冲突时，先确认用户运行的是哪一份东西。

## 这次最值得复用的方法论

如果以后再遇到“体验不对、感觉很慢、明明改了却没生效”的情况，我会优先按下面的顺序处理。

### 1. 用户反馈优先于本地自信

用户不是在帮你描述 bug，他们在帮你指出“你的真实系统边界在哪”。

### 2. 先确认运行路径，再确认代码路径

优先确认：

- 源码版还是打包版
- 当前进程还是旧托盘实例
- 本地构建产物还是 GitHub Release 产物
- 新包还是旧包

### 3. 把问题拆成可计时阶段

至少拆成这些阶段：

- 搜索
- 播放解析
- 首次缓冲 / loadedmetadata
- 歌词
- 颜色提取 / 非关键视觉链路

### 4. 结构性修复优先于局部 patch

优先思考：

- 能不能解耦
- 能不能换轻路径
- 能不能缓存
- 能不能把 fallback 从主路径挪出去

### 5. 修复后必须回到真实使用环境验证

不能只在：

- 单元测试里通过
- 本地源码环境里通过
- 自己机器的一条运行路径里通过

而是要回到用户真正点击的入口重新测。

## 一份以后可直接照着走的检查清单

## 当用户说“很慢”

1. 让问题先落到具体链路：搜索慢、播放慢、歌词慢、还是切歌慢
2. 确认用户打开的是源码、桌面包、安装版还是旧实例
3. 把整条链路分段计时
4. 找出最慢的那一段，而不是先讨论感受
5. 判断这是“重型主路径”问题，还是“旧产物”问题
6. 修复后回到同一入口复测

## 当用户说“明明改了但没生效”

1. 先查运行中的进程
2. 再查当前打开的文件路径
3. 再查构建产物生成时间
4. 再查 dist / release 是否重打
5. 最后才继续怀疑源码逻辑

## 当用户说“这个项目已经不是 MVP 了”

要检查的不是功能，而是这些条件是否成立：

- 数据可持久化
- 桌面端可长期使用
- 有安装包和发布路径
- 关键功能有回退方案
- 出现问题时能定位
- 文档可读

## 下一步优化建议

下面按“价值 / 成本 / 依赖关系”排序。

## 1. 个性化推荐与刷听流

### 价值

这是最符合产品定位的下一步，因为它能把“搜索型工具”变成“可沉浸消费的音乐产品”。

### 成本

中等偏高，需要行为数据、推荐策略和前端推荐流交互。

### 依赖

已经有收藏、历史、搜索记录和推荐接口雏形，是自然延伸。

## 2. 电台模式 / 情绪模式

### 价值

用户不一定总想搜歌，很多时候只是想进入某种氛围。

### 成本

中等，可以先用规则和标签做第一版，不一定要复杂算法。

### 依赖

依赖推荐接口和播放队列能力。

## 3. 播放列表与收藏体系升级

### 价值

从单曲收藏进化到“真正的个人音乐库”。

### 成本

中等，需要更完整的资料模型和前端管理界面。

### 依赖

依赖现有 SQLite 资料层，已经具备基础条件。

## 4. 下载资产管理

### 价值

一旦用户开始下载，软件就从在线播放器延伸成“本地音乐中心”。

### 成本

中等，需要下载管理、文件去重、元数据补全和离线入口。

### 依赖

依赖现有下载中心，属于可迭代增强。

## 5. 跨端同步

### 价值

长期使用价值会明显提升，尤其适合个人品牌产品。

### 成本

中高，需要轻量云端或同步服务。

### 依赖

依赖本地资料模型先稳定下来。

## 6. 诊断与自检面板

### 价值

能把这次最宝贵的排查经验产品化，让以后很多问题不需要靠人工对话才能定位。

### 成本

中等，需要把状态、缓存、provider、版本、运行路径可视化。

### 依赖

依赖现有 system-check 和日志能力，技术上门槛不高。

## 最后一句

如果要用一句话概括这次项目真正的进步，我会说：

> 我们不是把一个音乐软件“做出来了”，而是把一个原本靠运气能用的项目，变成了一个靠结构就能稳定工作的产品。
